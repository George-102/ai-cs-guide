# Distributed Systems

## What Is a Distributed System

Distributed systems study **how multiple computers work together to accomplish tasks that a single computer cannot**. Virtually every large-scale internet service you use — search engines, social networks, e-commerce, cloud storage — is powered by distributed systems.

The core challenges of distributed systems: networks are unreliable (messages can be lost, delayed, or arrive out of order), nodes can fail, clocks are not synchronized, and data must remain consistent across multiple replicas. Distributed system design is about finding workable compromises between these "impossibilities."

## Why Study Distributed Systems

**Required knowledge for large-scale systems.** Modern AI training (GPT, Gemini) requires thousands of GPUs working in coordination; AI inference services need deployment across multiple data centers globally — these are all distributed systems problems.

**Understand modern software architecture.** Microservices, container orchestration (Kubernetes), message queues, distributed databases — technologies you'll inevitably use in industry, all built on distributed systems theory.

**Advanced interview topic.** System Design Interviews are fundamentally about distributed system design. CAP theorem, consensus protocols, load balancing — these must-know topics for senior roles at major companies.

**Key to AI infrastructure.** Distributed training (data parallelism, model parallelism, pipeline parallelism), elastic model serving, edge AI deployment — AI engineers need to understand these distributed computing patterns.

## Core Knowledge Modules

### Distributed Systems Fundamentals

- **Definition and characteristics**: Resource sharing, concurrency, no global clock, independent failures
- **Architecture**: Client-server, P2P, microservices
- **Networking foundations**: RPC, messaging, TCP/IP review

### Time and Clocks

- **Clock synchronization**: Physical clocks, NTP protocol
- **Logical clocks**: Lamport clocks, vector clocks
- **Global state**: Chandy-Lamport snapshot algorithm

### Replication and Consistency

- **Replication strategies**: Primary-replica, multi-leader, leaderless
- **Consistency models**: Strong consistency, eventual consistency, causal consistency
- **CAP theorem**: The impossible triangle of Consistency, Availability, Partition tolerance
- **BASE theory**: Basically Available, Soft state, Eventually consistent

### Consensus Algorithms

- **Paxos**: The classic distributed consensus algorithm (hard to understand but important)
- **Raft**: The "understandable version" of Paxos, widely used in industry
- **ZAB**: Atomic broadcast protocol used by ZooKeeper
- **Two-Phase Commit (2PC) and Three-Phase Commit (3PC)**

### Distributed Data Storage

- **Data partitioning (Sharding)**: Hash partitioning, range partitioning, consistent hashing
- **Distributed databases**: Google Spanner, CockroachDB, TiDB
- **Distributed file systems**: GFS, HDFS
- **Distributed caching**: Redis Cluster, Memcached

### Distributed Computing

- **MapReduce**: Programming model for large-scale data processing
- **Spark**: In-memory computing engine, evolution of MapReduce
- **Distributed training**: Data parallelism, model parallelism, pipeline parallelism

### Fault Tolerance and Reliability

- **Failure models**: Crash failures, Byzantine failures
- **Replication and fault tolerance**: State machine replication
- **Distributed transactions**: Saga pattern, TCC

## How to Learn

**Read classic papers first.** Growing in distributed systems is inseparable from classic papers. Start with Google's "big three" (GFS, MapReduce, Bigtable), then move on to Raft, Spanner — each paper solves a real distributed problem.

**Do the MIT 6.824 labs.** This course has you implement the Raft consensus algorithm and a Raft-based KV store. After these labs, your understanding of distributed systems will surpass most practitioners.

**Start with consistent hashing.** Consistent hashing is one of the most elegant concepts in distributed systems, and the key to understanding data partitioning and load balancing.

**Build a distributed cluster with Docker Compose.** Deploy a Redis Cluster or Kafka cluster yourself and observe how the system recovers when nodes fail.
