# Database Systems

## What Is a Database System

A database system studies **how to efficiently, reliably, and securely store and retrieve large amounts of data**. From your phone's contact list to bank transaction records, from e-commerce inventory to social network graphs — virtually every piece of software has a database behind it.

Modern database systems do more than just "store data." The core problems they must solve include: how to find data quickly (indexing), how to prevent conflicts during concurrent operations (transactions and locks), how to avoid data loss during system failures (durability and recovery), and how to maintain high performance at scale (query optimization).

## Why Study Database Systems

**The foundation of backend development.** Whether it's web development, microservices, or AI applications, data access is an inescapable requirement. Without understanding databases, you can only build "toy-level" applications.

**Understanding the nature of data storage.** Vector databases (which you use in your RAG projects) are essentially databases too, except the index structure changes from B+ trees to approximate vector search. Understanding traditional databases makes learning vector databases much easier.

**Interview essentials.** Transaction ACID properties, indexing principles, SQL optimization, database design — these are guaranteed backend interview questions.

**Core subject in CS entrance exams.** Tests relational models, SQL, transaction processing, index structures, and more.

**Data layer for AI systems.** Feature stores, model metadata management, training data pipelines — AI engineering infrastructure relies heavily on databases.

## Core Knowledge Modules

### Relational Model and SQL

- **Relational model**: Tables, rows, columns; relational algebra (selection, projection, join, union/difference/cartesian product)
- **SQL**: DDL (CREATE TABLE), DML (INSERT/UPDATE/DELETE/SELECT), DCL (GRANT/REVOKE)
- **Advanced SQL**: Subqueries, aggregate functions (GROUP BY/HAVING), window functions
- **Database design**: ER diagrams, normalization (1NF through BCNF), denormalization

### Storage and Indexing

- **Disk storage**: A page (Page) is the basic unit of disk I/O
- **B+ tree index**: The most common database index structure, a multi-way balanced tree suited for range queries
- **Hash index**: O(1) exact lookups, no range query support
- **Clustered vs. non-clustered indexes**: Whether data is physically stored in index order
- **Composite indexes and leftmost prefix matching**
- **Index selectivity**: Only high-selectivity columns are good index candidates

### Query Processing and Optimization

- **Query execution plan**: The `EXPLAIN` command, understanding how the database executes your SQL
- **Join algorithms**: Nested loop join, sort-merge join, hash join
- **Query optimizer**: Cost-Based Optimization (CBO)
- **SQL tuning**: Avoid full table scans, use indexes properly, avoid SELECT *

### Transaction Processing

- **ACID properties**: Atomicity, Consistency, Isolation, Durability
- **Concurrency control**: Locks (shared/exclusive), Two-Phase Locking (2PL), deadlock detection and prevention
- **Isolation levels**: Read Uncommitted, Read Committed, Repeatable Read, Serializable
- **MVCC**: Multi-Version Concurrency Control, how InnoDB implements it

### Crash Recovery

- **Logging**: Write-Ahead Logging (WAL) — log first, then write data
- **ARIES algorithm**: The standard recovery algorithm in modern databases
- **Checkpointing**: Speeding up the recovery process
- **Backup strategies**: Full backup, incremental backup, differential backup

### NoSQL and NewSQL

- **Key-value stores**: Redis, DynamoDB — simple and efficient, great for caching and sessions
- **Document databases**: MongoDB — flexible schema, great for semi-structured data
- **Column-family stores**: Cassandra, HBase — suited for large-scale distributed writes
- **Graph databases**: Neo4j — designed for relationship queries (social networks, knowledge graphs)
- **NewSQL**: TiDB, CockroachDB — SQL consistency with NoSQL scalability

## How to Learn

**Balance theory and practice.** Database courses have many concepts and require heavy practice. Reading without writing SQL means concepts stay on paper forever.

**Start with a specific database.** Recommend MySQL first — great documentation, huge community, widely used. Install MySQL, create tables, write queries, inspect execution plans.

**Do the CMU 15-445 labs.** This course has you implement a complete database system (Buffer Pool Manager, B+ tree index, query executor, concurrency control). After these labs, your understanding of databases will be transformed.

**Learn to read EXPLAIN output.** This is the most important debugging tool in database development. After writing any SQL query, use `EXPLAIN` to verify the execution plan matches expectations.
